**Goal:** Ship the `/agentic-swarm:architect` skill (Phases 0 & 1) — research-driven decomposition that
turns a one-line goal into a plan-then-confirm PLAN and **zero-context briefs that forward-couple the
named gate into the brief** (the headline novelty) — layered over the shipped `/agentic-swarm` safe-fan-out
rails, reusing `/loop` for the optional Phase-4 persistence back-edge.

**Architecture:** Orchestrator-worker (Arch 1) on a plan-execute-verify skeleton. The new skill owns
Phases 0/1 (orient/research → decompose → select skill+gate set → contract → brief synthesis) and the
plan-then-confirm checkpoint; it **references** (never duplicates) the existing
`skills/agentic-swarm/` rails for Phase 2 (safe fan-out) and `reference/loops.md` for Phase 4. The gate
*runner* + gate library are deliberately **out of scope** (v0.7.0); v0.6.0 only forward-couples gate
*criteria* (inlined into the brief) for the MVP gates that will ship in v0.7.

**Tech stack:** Markdown skills (`SKILL.md` + `reference/*.md`), frontmatter = `name`+`description`
only, zero-dep plugin. Verification = `claude plugin validate --strict` (both modes) + a structural
test (Python `pytest`, matching the existing `tests/` suite) + the existing privacy/hook tests staying green.

**Resolved mechanics (verified against `docs/claude-code/skills.md` §2):** a plugin skill's command =
its **directory name** (`skills/architect/` → `/agentic-swarm:architect`); frontmatter `name` is only the
display label. So the directory is **`skills/architect/`** (the design's `skills/swarm-architect/` was
illustrative — the converged command `/agentic-swarm:architect` is the firm part).

**Scope discipline:** v0.6.0 is **not** auto-released — land it as a reviewable PR; the version bump +
tag happen on approval (lockstep), like v0.5.0.

## Task 2: Phase 0 — orient/research + decompose + select + contract

**Files:**
- Modify: `skills/architect/SKILL.md` (Phase 0 section)

**Step 1 — Write Phase 0 guidance.** Concrete, do-this steps for the orchestrator BEFORE any fan-out:
1. **Research "what does GOOD look like here?"** — domain norms, constraints, prior art (use Context7 for
   any library/tool; web/docs for domain). Keep it bounded (a small read-only scout, not a mega-swarm).
2. **Classify the use-case** (web UI · game · CLI · API · data/ML · library · docs · audit · research).
3. **Decompose into workstreams** (the work-list) — each independently briefable, with a stable key.
4. **Select the skill-set + GATE-set per workstream** via `reference/usecase-gate-map.md`. **Anti-theater
   rule (load-bearing):** only select a gate that has shipped, checkable criteria; mark anything else
   `future / not-yet-built` and do NOT forward-couple it into a brief.
5. **Define the shared CONTRACT** (interfaces, design tokens, conventions, output schema) so independent
   outputs integrate.
6. Output: a **PLAN** = workstreams × skills × gates × contract.

**Step 2 — Skill-detection note.** Detecting which optional enhancer skills are installed is a zero-dep
filesystem probe (`~/.claude/skills/<id>/SKILL.md`, `.claude/skills/<id>/SKILL.md`,
`~/.claude/plugins/cache/**/skills/<id>/SKILL.md`) matching the on-disk **directory** name. External
skills are opportunistic enhancers only — never a hard dependency (portability).

**Step 3 — Verify validate (both modes).** Expected: pass.

**Step 4 — Commit:** `feat(architect): Phase 0 — research, decompose, skill/gate selection, contract`

**Step 1 — Write `reference/brief-template.md`.** A `/agent-prompt`-style **zero-context** worker brief
with these sections (the **gate forward-coupling is the headline novelty**): Mission (one goal) ·
Research context (what GOOD looks like, from Phase 0) · Contract/interfaces (shared) · Expected output

+ `assets` gate criteria inlined). The template must be **self-contained** — it inlines criteria so a
worker passes the gate with zero external skills.

gate-set selection feeds the "MUST PASS GATES" block.
