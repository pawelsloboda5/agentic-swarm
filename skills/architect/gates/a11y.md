# Gate: `a11y`

Run via [`../reference/gate-runner.md`](../reference/gate-runner.md). Mixed tier: an **objective floor =
a scoped automated runner sweep** (axe/pa11y/Lighthouse, browser-dependent) plus a separate-context
**static-semantics & keyboard critic**. This is the **full standalone** a11y gate (the cheap a11y checks —
contrast, alt text, accessible-name *presence*, tap-target — stay owned by [`ui-ux`](ui-ux.md); this gate
does **not** re-gate them). **PASS ≠ conformance** (see below).

## Definition

- **id:** `a11y`
- **applies_when:** the workstream renders HTML / web UI **and** a deeper-than-cheap a11y bar is wanted —
  distinct from `ui-ux`'s folded cheap a11y. Contrast / alt text / accessible-name presence / tap-target
  are **owned by [`ui-ux`](ui-ux.md)**; this gate adds the automated runner sweep + the keyboard/semantics
  critic on top, never duplicating them.
- **tier:** `mixed` (scoped objective runner floor + a separate-context critic rung)
- **criteria:**
  - **Objective floor — scoped automated runner sweep (browser-dependent; OPTIONAL enhancer):**
    1. **Auto-detect a runner** (priority `axe` → `pa11y` → `lighthouse`) and run it against the rendered
       output, then normalize its JSON with [`lib/a11y_report.py`](lib/a11y_report.py):
       `python gates/lib/a11y_report.py <runner-json> --runner axe|pa11y|lighthouse` (run from the skill
       root; **exit 0 = pass, 1 = gating violations, 2 = unreadable**). **Gate iff `gating_violations == 0`.**
    2. **Single-owner partition (the seam):** the normalizer's `UIUX_OWNED_RULES` / `UIUX_OWNED_SC`
       (`color-contrast`, the image-alt family, the accessible-name family: `label`/`button-name`/
       `link-name`/`aria-*-name`, and `target-size`; pa11y SCs `1_4_3`/`1_1_1`/`4_1_2`/`2_5_8`) are
       surfaced **advisory** (deferred to `ui-ux`) and **never gate here** — so a single contrast/alt
       defect never produces two failing verdicts. `axe` is preferred (rule ids partition cleanly; the
       pa11y SC-substring partition is approximate).
    3. **Lighthouse is advisory, never a hard gate** (its score is a weighted subset); a null score or a
       top-level `runtimeError` is `degraded`, never coerced to 0.
    4. **No runner present ⇒ SKIP this sub-check** with a visible note ("runner unavailable; automated
       a11y not run") — never a fail-because-absent, and **never a borrowed-contrast pass** (`ui-ux`
       owns contrast). The verdict then rests on the critic rung below.
  - **Critic rung — static-semantics & keyboard (separate context — the `verifier`):**
    5. **Statically checkable from the HTML source (these GATE, browser-free):** landmark + heading
       structure (one `main`, ordered headings, regions named), programmatic **form-label association**
       (control tied to its label, beyond mere name presence), **role / value / state ARIA** correctness
       (valid roles, required attrs, valid values — **excluding accessible-name presence, which `ui-ux`
       owns**), and DOM/reading order.
    6. **Render-dependent (downgrade to `advisory` when no browser):** actual keyboard **tab-order
       reachability** of every interactive control, and **visible-focus effectiveness** (distinct from
       `ui-ux`'s static `:focus-visible`-defined check — this judges the *rendered* indicator). Binary
       per-criterion, **one** grounded pass, citations.
- **verifier:** the critic prompt — in a fresh context, judge criteria 5–6 (semantics/keyboard), citing
  the offending selectors; the producer never grades itself; no re-score loop. The static parts (5) gate
  even browserless; the render parts (6) are advisory without a browser.
- **confidence:** **never `1.0` on automation alone** — axe/pa11y cover only ~30–50% of WCAG SC. High when
  a runner ran clean **and** the critic gated; **reduced + `degraded: true`** when no runner ran (static
  critic only); an honest **`flag`** when nothing a11y-specific is verifiable (no runner **and** the
  static critic cannot run) — never a `pass` off borrowed evidence.
- **backing_skill:** `responsive-ui-audit` (Playwright + axe), `wcag-aa-contrast-remediation`,
  `web-design-guidelines` — **optional enhancers only**; the bundled normalizer + static critic stand alone.

## Evidence required for a pass

The runner command + the `a11y_report.py` verdict (`gating_violations == 0`, the partitioned advisory
count, `incomplete_count`) **and** the critic's per-criterion booleans with cited selectors. A holistic
"looks accessible" with no evidence is not a pass (anti-theater invariant).

## PASS ≠ conformance

An automated green run + a passing critic is **necessary, not sufficient**: automated tools detect only
~30–50% of WCAG success criteria, and the critic samples the rest. A `pass` means "no automated violation
in scope and the sampled semantics/keyboard checks held" — **not** full WCAG conformance, which needs a
manual audit (screen-reader pass, real keyboard walk). Confidence therefore never reaches `1.0` on
automation alone.

## On fail

Re-brief with the exact unmet criteria (e.g. "axe `landmark-one-main` fails"; "input #email has no
programmatic label"; "button #play unreachable by keyboard"), bounded N=2; then honest `flag` listing
`unmet_criteria`. No runner ⇒ note "automated a11y not run; cheap checks owned by `ui-ux`" and gate on the
static critic alone; if even that cannot run ⇒ `flag` ("cannot objectively verify"), never a silent pass.
Advisory items (ui-ux-owned violations, Lighthouse score, render-dependent checks without a browser) never
re-run.
