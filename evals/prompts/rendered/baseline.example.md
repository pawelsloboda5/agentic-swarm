# Claude Code `Workflow` script API (reference)

A Workflow script is plain JavaScript that orchestrates subagents. It runs in an async context
(use `await` directly). It must begin with an `export const meta = {...}` literal, then a script
body that uses the hooks below.

```js
export const meta = {
  name: 'my-workflow',
  description: 'one-line description',
  phases: [{ title: 'Find' }, { title: 'Synthesize' }],
}
// body starts here — await is allowed at top level
```

## Hooks available in the script body

- `agent(prompt: string, opts?): Promise<any>` — spawn one subagent. Without a `schema` it
  resolves to the subagent's final text. With `opts.schema` (a JSON Schema) it resolves to the
  validated object. **It resolves to `null`** if the subagent is skipped or dies on a terminal
  API error after the runtime's own internal retries. `opts` may include: `label`, `phase`,
  `schema`, `model`, `effort` ('low'|'medium'|'high'|'xhigh'|'max'), `isolation: 'worktree'`,
  `agentType`.
- `parallel(thunks: Array<() => Promise<any>>): Promise<any[]>` — run thunks concurrently and
  await **all** of them (a barrier). A thunk that throws (or whose agent errors) resolves to
  `null` in the returned array; the call itself never rejects.
- `pipeline(items, stage1, stage2, ...): Promise<any[]>` — run each item through all stages
  independently, with **no barrier between stages** (item A can be in stage 3 while item B is in
  stage 1). Each stage callback receives `(prevResult, originalItem, index)`. A stage that throws
  drops that item to `null` and skips its remaining stages.
- `phase(title: string): void` — start a named phase (groups subsequent agents in the UI).
- `log(message: string): void` — emit a progress line to the user.
- `args: any` — the value passed as the Workflow tool's `args` input.
- `budget: { total: number|null, spent(): number, remaining(): number }` — the turn's output-token
  target (or `total === null` if none). `agent()` throws once `spent()` reaches `total`.
- `workflow(nameOrRef, args?): Promise<any>` — run another workflow inline (one level deep only).

## Scheduling a future wake-up (available from the surrounding agent, not the script body)

`ScheduleWakeup({ delaySeconds: number, reason: string, prompt: string })` schedules the agent to
be re-invoked later with `prompt`. `delaySeconds` is clamped to `[60, 3600]`.

## Runtime facts

- Concurrent `agent()` calls are capped at `min(16, cpu_cores - 2)` per workflow; excess calls
  queue. A single `parallel()`/`pipeline()` call accepts at most 4096 items; lifetime agent count
  is capped at 1000.
- The completion of a workflow produces a `<task-notification>`.
- The script sandbox blocks `Date.now()`, `Math.random()`, and argless `new Date()` (they would
  break deterministic resume), and may block timers.
- Resume: relaunching with `Workflow({ scriptPath, resumeFromRunId })` returns cached results for
  `agent()` calls whose `(prompt, opts)` are unchanged; the first changed/new call and everything
  after it re-runs. Every invocation persists its script to a file and returns the `scriptPath`
  and a `runId`.
- Standard JS built-ins are available; there is no filesystem/network access from the script body.
