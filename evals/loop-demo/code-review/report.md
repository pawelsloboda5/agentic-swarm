## HIGH (1)

### Unhandled HTTP/2 session error can crash the process
- **File / line:** `lib/helpers/Http2Sessions.js` :41, :107
- **Issue:** No `'error'` listener is attached to the pooled `ClientHttp2Session`. http2 sessions emit `'error'` on TLS failures, GOAWAY, and socket resets; an EventEmitter with no `'error'` handler re-throws and crashes the whole Node process. Pooled sessions outlive any single request, so an async error on an idle/cached session has no consumer to catch it — a single bad connection can take down the host application.
- **Fix:** Attach `session.once('error', removeSession)` (or a logging/cleanup handler) alongside the existing `'close'` handler so a session error tears down the pool entry instead of propagating fatally.

## MEDIUM (29)

### Resource leaks & lifecycle/cleanup
These share a root cause: listeners/subscriptions/timers registered on one path are not symmetrically released on every exit path.

- **`lib/helpers/Http2Sessions.js` :82-105** — A session created with `sessionTimeout` but on which `session.request` is never called leaks: the idle timer is only armed after a stream's `'close'`. If a caller gets a session then abandons it, `streamsCount` stays 0 and the open session is never reclaimed. *Fix:* arm an initial idle timer right after session creation, cleared on first request.
- **`lib/helpers/Http2Sessions.js` :96-99** — The idle `setTimeout` is not `unref()`'d, so a pending idle-close timer keeps the Node event loop alive (delaying clean exit by up to `sessionTimeout`). *Fix:* `timer.unref()`.
- **`lib/adapters/fetch.js` :458-462, :538** — Stream-response path leaks the composed-signal subscription (timeout timer + abort listeners). When `isStreamResponse` is true but `response.body` is falsy, the wrapping block is skipped so `unsubscribe` is never registered, and line 538's `!isStreamResponse && ...` guard prevents cleanup. The success path returns with the timeout/abort listeners dangling. *Fix:* ensure `unsubscribe` runs for stream responses with absent bodies.
- **`lib/cancel/CancelToken.js` :105-117** — `toAbortSignal` subscribes `abort` but nothing auto-unsubscribes on normal request completion; the AbortController/signal stays referenced for the token's whole life, accumulating on long-lived/reused tokens. *Fix:* auto-teardown on settle, or have axios call `signal.unsubscribe` on request settle.
- **`lib/cancel/CancelToken.js` :39-52** — The overridden `promise.then` calls `token.subscribe(resolve)` per invocation and only reaps it if the returned `promise.cancel()` is called. Any code that merely awaits/`.then`s the token adds a permanent `_listeners` entry — unbounded growth on a reused token. *Fix:* unsubscribe when the inner promise settles.
- **`lib/cancel/CancelToken.js` :32-35** — A throwing listener aborts notification of all remaining subscribers: the `while (i-- > 0)` loop has no try/catch, so one throw terminates the loop, leaves `_listeners` un-nulled (leak), and surfaces as an unhandled rejection inside `Promise.then`. *Fix:* wrap each invocation in try/catch; null `_listeners` in `finally`.

### Cancellation / abort correctness
- **`lib/helpers/composeSignals.js` :48 (+ missing guard ~8-12)** — Already-aborted input signals are not honored: subscription is via `addEventListener('abort', …)` only, which never fires for a signal already aborted before attach, so the composed controller never aborts and the request proceeds despite cancellation. *Fix:* check `signals.some(s => s.aborted)` and invoke `onabort` immediately.
- **`lib/adapters/xhr.js` :204-207, :226** — Pre-aborted signal path falls through to `request.send()` on a nulled request: `onCanceled()` runs synchronously and sets `request = null`, but execution continues to `request.send(...)`, throwing a TypeError in the Promise executor. *Fix:* `return` after `onCanceled()` for an already-aborted signal (or guard the `send` with `if (request)`).

### Request routing / proxy / SSRF-adjacent
- **`lib/adapters/http.js` :286-288, :373-375** — On redirect, `setProxy()` strips the CONNECT-tunneling agent only from `options.agent`, but follow-redirects consumes `options.agents.https`, which still references the stale tunneling agent. A non-proxied redirect target is then sent through a proxy tunnel that no longer applies. *Fix:* also clear `options.agents.http/https` when the bound agent carries the `kAxiosInstalledTunnel` marker.
- **`lib/helpers/shouldBypassProxy.js` :16-39, :208** — Over-broad loopback equivalence: `isLoopback` folds `localhost`, `127.0.0.0/8`, `::1`, AND the unspecified `0.0.0.0`/`::` together, so `NO_PROXY=localhost` bypasses the proxy for all of them — including `0.0.0.0`/`::`, which route to local services. If a proxy is the security boundary, one loopback entry silently opens all forms. *Fix:* reconsider classing unspecified addresses as loopback; make symmetric folding explicit/configurable.
- **`lib/helpers/shouldBypassProxy.js` :174, :200-201** — `'*'` only works as the entire `NO_PROXY` value, not as a list entry (`NO_PROXY="foo,*"` or `" * "` bypasses nothing), diverging from curl's match-all-anywhere semantics. *Fix:* treat a bare `'*'` token within the split list as match-all.

### Memory / DoS — body sizing
- **`lib/helpers/fromDataURI.js` :52-54** — No bound on decoded payload size: `Buffer.from(body, …)` allocates proportional to URI length and the regex body group accepts unlimited input — a large `data:` URI (e.g. from a redirect Location or user input) causes unbounded memory allocation. *Fix:* enforce a max size (e.g. `options.maxContentLength`) before allocating.
- **`lib/adapters/fetch.js` :168-173, :312-318** — `maxBodyLength` enforcement for spec FormData fully materializes the body (`new Request(...).arrayBuffer()`) before the limit check, so an oversized form is buffered into memory (doubling resident memory for blob-backed forms) before rejection — an OOM/DoS vector. *Fix:* stream/per-chunk size count, or cap measurement at `maxBodyLength+1`.
- **`lib/helpers/estimateDataURLDecodedBytes.js` :73-75** — Unpadded base64 underestimates decoded size (`Math.floor(effectiveLen/4)` drops the final partial group). For a size guard, underestimation is the unsafe direction — lets slightly-oversized payloads through. *Fix:* add partial-group bytes for remainders of 2/3 chars, or require canonical padding.
- **`lib/helpers/estimateDataURLDecodedBytes.js` :46-74** — Padding/`effectiveLen` logic assumes well-formed base64 and can drive the result negative → clamped to 0, so a crafted/malformed body can under-report its size to the guard. *Fix:* only subtract padding when `groups>0` and `effectiveLen%4===0`; treat invalid padding conservatively (round up).

### Error contract / consistency
- **`lib/helpers/fromDataURI.js` :54** — Non-base64 path runs `decodeURIComponent(body)`; a malformed percent sequence throws a raw `URIError` that escapes uncaught, violating the function's "always throw a typed AxiosError" contract. *Fix:* wrap in try/catch and rethrow `AxiosError.ERR_INVALID_URL`.
- **`lib/helpers/formDataToStream.js` :33, :35, :46-47** — Typed-array part values silently drop the `Content-Length` header: `encode()` supports typed arrays but `contentLength` uses `value.size` (a Blob-only prop), yielding `undefined` → `NaN` → omitted header → forced chunked transfer with no declared length. *Fix:* use `value.byteLength` for typed arrays, mirroring the `encode()` branching.

### Interceptor / dispatch correctness
- **`lib/core/Axios.js` :208-223** — Synchronous interceptor chain dispatches anyway after a failed request interceptor: when `onFulfilled` throws and `onRejected` does not rethrow, the loop `break`s but falls through to `dispatchRequest`, sending a request the interceptor intended to block/transform with pre-failure config. *Fix:* short-circuit (return rejected promise / rethrow) after `onRejected`.
- **`lib/core/dispatchRequest.js` :55-60, :72-81** — `config.response = response` mutates the caller-supplied config and is not reentrancy/concurrency safe: two concurrent `axios(config)` calls (or a re-entrant `transformResponse`) clobber each other's `config.response`, so an AxiosError can attach the wrong response; the `finally` `delete` also destroys any legitimately pre-existing `response` key. *Fix:* pass the response explicitly (transformData already receives it) or use a per-call context object.

### Data integrity / serialization
- **`lib/core/AxiosError.js` :54-57** — Redaction bypass for non-plain objects exposing `toJSON`: such objects are returned verbatim without descending, so redact-listed keys nested inside class instances (e.g. credential wrappers) leak into the serialized snapshot. *Fix:* recurse into the `toJSON()` output and apply redaction.
- **`lib/core/AxiosError.js` :56** — `return source` returns a live reference (not a copy) for `toJSON`-bearing objects, so the "snapshot" aliases live config internals (sockets, agents, streams), defeating the snapshot intent and risking mutation/circular issues at stringify time. *Fix:* return a cloned/serialized form.
- **`lib/core/AxiosHeaders.js` :304-309** — `static accessor()` has a broken lazy-init: `this[$internals] = this[$internals] = { accessors: {} }` self-assigns instead of using a `|| {…}` fallback, so every call discards existing internals and resets the dedup cache — `buildAccessors` re-runs/redefines properties on every invocation. *Fix:* `const internals = this[$internals] = (this[$internals] || { accessors: {} });`
- **`lib/utils.js` :686-697** — `forEachEntry`'s `obj && obj[iterator]` guard is ineffective: if `generator` is falsy, the next line unconditionally calls `generator.call(obj)`, throwing `TypeError: Cannot read properties of undefined`. *Fix:* `if (!isFunction(generator)) return;` first.
- **`lib/utils.js` :707-716** — `matchAll` infinite-loops if passed a non-global RegExp: `regExp.exec(str)` keeps matching from index 0 and `lastIndex` never advances, hanging the process. *Fix:* assert/normalize the `g` flag, use native `String.prototype.matchAll`, or break when `index` doesn't advance.

### Progress / auth / throttle behavior
- **`lib/helpers/progressEventReducer.js` :13-19, :24** — `loaded` is silently clamped to `total` and `bytesNotified` is monotonically pinned, so when actual transferred bytes exceed declared `total` (chunked/gzip, inaccurate Content-Length) progress freezes at 100% while data still flows and the speedometer is fed 0-byte samples that decay the reported rate. *Fix:* clamp only for the progress math; report real `rawLoaded` to consumers.
- **`lib/helpers/resolveConfig.js` :66-73** — Basic-auth username is NOT passed through `encodeUTF8` like the password, so a username with any non-Latin-1 char makes `btoa()` throw `InvalidCharacterError` (rethrown as `ERR_BAD_OPTION_VALUE`), rejecting otherwise-valid credentials. *Fix:* `btoa(encodeUTF8(username) + ':' + (password ? encodeUTF8(password) : ''))`.
- **`lib/helpers/throttle.js` :9, :26, :34** — No validation of `freq`: `freq <= 0` disables throttling (every call fires), and `0`/`NaN` produce Infinity/NaN that coerce the leading call into a deferred macrotask. *Fix:* validate `freq` is a finite positive number at construction.
- **`lib/helpers/throttle.js` :30-34, :39** — No way to cancel a pending trailing call: the scheduled timer still fires after the consumer stops using the function (e.g. teardown), causing a late/stale invocation and a lingering timer. *Fix:* expose a `cancel` that clears `timer`/`lastArgs`.

### Module-init robustness
- **`lib/adapters/adapters.js` :30-34** — Asymmetric prototype-pollution hardening: the `name` `defineProperty` uses `__proto__: null` and is wrapped in try/catch, but the adjacent `adapterName` `defineProperty` is neither, so a polluted `Object.prototype.get` throws uncaught during module init, breaking axios entirely — defeating the intended protection. *Fix:* give line 34 the same `__proto__: null` descriptor inside the same try/catch.

## Cross-file themes

1. **Asymmetric register/unregister & timer lifecycle (the dominant theme).** The same defect recurs: a listener, subscription, or timer is set up on the happy path but not released on every exit. Seen in `Http2Sessions.js` (error/idle/unref), `fetch.js` (stream unsubscribe), `CancelToken.js` (×3 — throwing listener, toAbortSignal, promise.then), `composeSignals.js` (add vs remove via different APIs), `xhr.js` (no listener removal), `throttle.js` (no cancel), and `AxiosTransformStream.js` (untracked setTimeout). Several are flagged MEDIUM as functional leaks; many more are LOW. A shared "subscribe returns an unsubscribe; always call it in finally" pattern, plus `unref()` on every library-created timer, would eliminate a whole class of these.

2. **Inconsistent prototype-pollution hardening.** The codebase deliberately hardens against `Object.prototype` pollution in many places (`Object.create(null)`, `__proto__: null` descriptors, `utils.hasOwnProp`, null-proto config in `mergeConfig`), but applies it unevenly: `adapters.js` (one of two `defineProperty` calls unguarded), `parseHeaders.js` (`__proto__` header silently lost via prototype read), `formDataToJSON.js` (only `__proto__` denylisted, relying on incidental overwrite for `constructor`/`prototype`), `AxiosHeaders.js` (header *names* not CRLF/validity-checked while values are). The intent is right; the application has gaps.

3. **Body/payload size guards are approximate or bypassable.** `estimateDataURLDecodedBytes.js` (under-counts in the unsafe direction), `fromDataURI.js` (no bound at all), and `fetch.js` (materializes before measuring) all weaken the size/DoS protections they exist to provide. Size limits should measure conservatively (round up / cap at limit+1) and reject before allocation.

4. **Error-contract inconsistency.** Throw sites diverge on whether they emit a typed `AxiosError` with a `.code`: `adapters.js` ("Unknown adapter"), `toFormData.js` (Blob-unsupported, circular ref → plain Error), and `fromDataURI.js` (raw URIError escapes) all break the otherwise-consistent typed-error convention, so consumers' `err.code` / `instanceof AxiosError` checks miss these paths.

5. **Mutation of caller-supplied objects.** `dispatchRequest.js` (`config.response`), `Axios.js` (`delete headers[method]` in place), and `toFormData.js` (module-level `predicates` Object.assign) all mutate shared/caller state, creating reentrancy and reuse hazards. Operate on per-call copies.

6. **Heuristics that misclassify edge inputs.** Single-byte/dot-presence shortcuts misfire: `http.js` `resolveFamily` (dot → IPv4 misclassifies `::ffff:1.2.3.4`), `ZlibHeaderTransformStream.js` (1-byte zlib detection), `fetch.js` `maybeWithAuthCredentials` (bare `:` over-matches), and `shouldBypassProxy.js` (loopback/`*`/leading-dot semantics). Individually low-to-medium, collectively a reminder to validate against the full input grammar.
